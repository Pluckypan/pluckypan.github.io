一张图片引发的血案

ImageBrowser图片浏览控件 如何打造一个支持 动图、普通图、超大图 并支持 缩放 另外支持 过渡动画 的图片浏览控件

故事的开始

看似简单

看到以上的需求，第一感觉 简单(只有这些确实很简单)

  1. Don + ViewPager 自定义
  2. Glide 显示图片
  3. 过渡动画使用 ViewAnimator
  4. 6~8 小时开发工作量

第一波坑: 需要支持缩放平移

从需求上看，根本没有提及手势识别相关的功能，但是主流的图片浏览器都支持缩放，问了问产品和iOS,加！Android 官方提供支持手势识别类，实现也不难。(GooglePhotosGestureDetector)

手势识别

基于 GestureDetector 实现图片平移、双击缩放、单击

GestureDetector 简介
  1. 接口
    • OnGestureListener 6种最基本的事件 onDown onShowPress onSingleTapUp onScroll onLongPress onFling
    • OnDoubleTapListener 双击相关接口 onSingleTapConfirmed onDoubleTap onDoubleTapEvent
    • OnContextClickListener 外接设备接口 onContextClick
    • SimpleOnGestureListener 静态外部类 默认实现了以上三个接口 根据需要重新对应的接口即可
  2. 创建并绑定接口 GestureDetector mGestureDetector=new GestureDetector(context,listener)
  3. 事件分发 自定义View的 onTouchEvent 交给处理 mGestureDetector
     @Override
 public boolean onTouchEvent(MotionEvent event) {
     return mGestureDetector.onTouchEvent(event);
 }
  4. 拿 onDown 接口为例 只有返回值为 true 才会触发接下来的一系列接口
    /*
 *
 * Notified when a tap occurs with the down {@link MotionEvent}
 * that triggered it. This will be triggered immediately for
 * every down event. All other events should be preceded by this.
 *
 * @param e The down motion event.
 */
boolean onDown(MotionEvent e);
  5. 手势总结
    • 快速点击: onDow() -> onSingleTapUp() -> onSingleTapConfirmed()
    • 长按不滑动: onDown() -> onShowPress() -> onLongPress()
    • 短按不滑动: onDown() -> onShowPress() -> onSingleTapUp() -> onSingleTapConfirmed()
    • 拖动: onDown() -> onScroll() -> onScroll()-> ...(如果拖动释放时故意甩手 会触发 onFling)
    • 惯性滑动: onDown() -> onScroll() -> .... -> onFling()
    • 快速点击两下: onDown() -> onSingleTapUp() -> onDoubleTap() -> onDoubleTapEvent() -> onDown(?) -> onDoubleTapEvent()-> ...
    • onDown 返回值为 false : onDown() -> onShowPress() -> onLongPress() 与长按事件一样 不太合理

      查看源代码原因为 默认行为 LONGPRESS_TIMEOUT=500 TAP_TIMEOUT=100

缩放识别

基于 ScaleGestureDetector 实现图片双指缩放

ScaleGestureDetector 简介
  1. 接口 OnScaleGestureListener
    public interface OnScaleGestureListener {
     //如果为true则表示缩放是否已经达成 如果达成则不继续 如果为false 则会一直进行：应用场景是控制缩放比例  比如小于0.5被则不继续缩放 只要返回true即可
     public boolean onScale(ScaleGestureDetector detector);
     //如果返回true 则表示继续缩放事件 false 则终止缩放事件  应用场景是：在特定的区域才支持缩放、比如苹果的触控板区域才支持缩放
     public boolean onScaleBegin(ScaleGestureDetector detector);
     public void onScaleEnd(ScaleGestureDetector detector);
 }
  2. 创建并绑定接口 ScaleGestureDetector mGestureDetector=new ScaleGestureDetector(context,listener)
  3. 事件分发 自定义View的 onTouchEvent 交给处理 mGestureDetector
     @Override
 public boolean onTouchEvent(MotionEvent event) {
     return mGestureDetector.onTouchEvent(event);
 }
  4. 对外常用方法
    //缩放是否正处在进行中
public boolean isInProgress();
//返回组成缩放手势(两个手指)中点x的位置
public float getFocusX();
//返回组成缩放手势(两个手指)中点y的位置
public float getFocusY();
//获取本次缩放事件的缩放因子,缩放事件以onScale()返回值为基准，一旦该方法返回true，代表本次事件结束，重新开启下次缩放事件。
public float getScaleFactor();
  5. 其他方法
    //组成缩放手势的两个触点的跨度(两个触点间的距离)
public float getCurrentSpan();
//同上，x的距离
public float getCurrentSpanX();
//同上，y的距离
public float getCurrentSpanY();

手势识别整合

两种手势本是相互干扰的，如果 ScaleGestureDetector 正在进行 但 onTouchEvent 事件突然被打断会引发后续状态不对 实现方式有两种

多点触控才启用缩放 缩放时有跳动
@Override
public boolean onTouchEvent(MotionEvent event) {
    int pointCount = event.getPointerCount();
    if (pointCount > 1) {
        return mScaleDetector.onTouchEvent(event);
    } else {
        return mNormalDetector.onTouchEvent(event);
    }
}
如果没有在缩放 则切换为普通手势监听 体验更佳
@Override
public boolean onTouchEvent(MotionEvent event) {
    boolean res = mScaleDetector.onTouchEvent(event);
    if (!mScaleDetector.isInProgress()) {
        res = mNormalDetector.onTouchEvent(event);
    }
    return res;
}

整合图片缩放与过渡动画

  1. 如果借助Activity或者Fragment使用共享元素动画,不用处理任何逻辑就能实现,图片被放大都返回列表可以正常缩小并与列表图片完全吻合，动画很自然流畅 演示DEMO。但是存在以下问题：

    • Activity 不同Window绘制 焦点变换会触发 虚拟导航栏和状态栏弹出，Fragment形式需要 两个Fragment之间replace才能实现动画
  2. 踩过这些坑之后,决定学学 微信 怎么实现的，Github 搜索微信图片
    • 大部分都采用 PhotoView
    • 基于Activity 或者 Dialog,不满足条件(Dialog 可以去掉Windows的Focus Flag 但是会影响点击事件)
    • 根据PhotoView 关键字去搜索 找到了与我们设计契合的项目 imagebrowser
  3. 改造 imagebrowser
    • 自定义加载器保留
    • PhotoView 图片过渡动画(图片放大后退出时 Matrix同步刷新保持与原图吻合)
    • 主体设计框架保留，但是不能直接基于DecorView,因为滑出虚拟导航栏时 图片显示异常
    • 删除一些无用代码、解决一些Crash。
  4. 至此，一个和朋友圈差不多的图片浏览器基本实现

第二波坑：GIF 和 大图

由于需求分工不属于同一个人，图片选择及发布的功能比较靠后，后面有出现 上传大图或者动图的情况。动图很简单，Glide本身支持，但是大图就麻烦了，因为需要解决 PhotoView 遇到一切问题。

  1. 秀秀的方案是屏蔽大图 Wide第一版本采用的方案
  2. Glide默认配置加载大图 模糊到看不清(微博大图看不清文字)
  3. 放开Glide的压缩配置 直接Glide.into SimpleTarget,结论 卡,OOM
  4. ImageBrowser 现有的设计明显不能满足当前的需求，之前只有图片加载器直接显示图片，现在必须要先下载原始图片才能明确这图是不是大图，于是添加了下载器的概念。
  5. 有了下载好的图片之后，能判断是否大图后，选择出现，要在PhotoView的基础上改造以支持大图区域加载还是，直接引入 subsampling-scale-image-view算逑（之前用过的）
  6. 带着这些疑问 进入 番外篇

番外篇之Bitmap

  1. 图片占用内存计算
    • 计算公式：图片宽图片高单位像素所占字节数
    • Bitmap几种解码格式 ARGB 0~255
像素格式 单位像素字节数 备注
ARGB_8888 4 Byte 默认 32位 4字节 保持色彩真实
RGB_565 2 Byte 丢失精度 失真 但是效果差异可接受,不支持alpha通道
ARGB_4444 2 Byte 支持alpha通道 但是失真较严重 Android已不建议使用
ALPHA_8 1 Byte 只保留alpha通道,比如黑白透明度图片
  • 10241024 像素 RGB_565 分配内存 为 10241024*2=2MB 而ARGB_8888 为4MB，可节省一半的内存
    1. RGB_565 和 ARGB_4444 是如何节省内存的 详见图片
    2. option.inPreferredConfig 一定会起作用吗? ` /**
    • If this is non-null, the decoder will try to decode into this
    • internal configuration. If it is null, or the request cannot be met,
    • the decoder will try to pick the best matching config based on the
    • system's screen depth, and characteristics of the original image such
    • as if it has per-pixel alpha (requiring a config that also does). *
    • Image are loaded with the {@link Bitmap.Config#ARGB_8888} config by
    • default. 详见DEMO
    • 结论: 系统会根据图片以及系统的支持情况来确定到底使用哪种 解码格式。
    • 一般来讲，对于jpg格式图片，使用RGB_565格式进行解码，图片占用的内存会更少 */ `
    1. Bitmap.compress 压缩图片 会节省内存吗?
      int len = bm.getByteCount(); // 原始图片的byte[]数组大小
ByteArrayOutputStream out = new ByteArrayOutputStream();
bm.compress(Bitmap.CompressFormat.JPEG, 80, out);
int compressedLen = out.toByteArray().length; // 这里out.toByteArray()所返回的byte[]数组大小确实变小了！
Bitmap compressedBm = BitmapFactory.decodeByteArray(out.toByteArray(), 0, compressedLen);
int newLen = compressedBm.getByteCount(); // 这里decode出来的图片byte[]数组大小和原始图片是一样的！并没有变小
//结论是 会节省磁盘空间 但是并不节省内存空间  因为内存分配公式为 图片长*图片宽*单位像素所占字节数
    2. 节省内存的措施 Bitmap 采样率

int scale = Math.round(1 / mRatio);
options.inSampleSize = scale > 1 ? scale : 1;

按区域分块加载Bitmap

了解以上关于Bitmap的基础知识之后，我们接下来看看如何通过 BitmapRegionDecoder 加载大图

BitmapRegionDecoder简介

Create a BitmapRegionDecoder from the specified byte array. Currently only the JPEG and PNG formats are supported. 仅仅279行代码，很简单(因为都是Native在处理)

  1. 4个静态构造方法：
    • public static BitmapRegionDecoder newInstance(byte[] data,int offset,int length, boolean isShareable)
    • public static BitmapRegionDecoder newInstance(FileDescriptor fd, boolean isShareable)
    • public static BitmapRegionDecoder newInstance(InputStream is,boolean isShareable)
    • public static BitmapRegionDecoder newInstance(String pathName,boolean isShareable)
  2. 说明必须要先拿到图片(JPG、PNG)才能处理
  3. isShareable 理解? SubsamplingScaleImageView 为false
    /**
* @param isShareable If this is true, then the BitmapRegionDecoder may keep a
*                    shallow reference to the input. If this is false,
*                    then the BitmapRegionDecoder will explicitly make a copy of the
*                    input data, and keep that. Even if sharing is allowed,
*                    the implementation may still decide to make a deep
*                    copy of the input data. If an image is progressively encoded,
*                    allowing sharing may degrade the decoding speed.
*/
  4. 对外方法
    • public Bitmap decodeRegion(Rect rect, BitmapFactory.Options options);//耗时
    • public int getWidth()
    • public int getHeight()
    • public void recycle()
    • public final boolean isRecycled()

演示DEMO

  1. 图片分辨率计算 inJustDecodeBounds
     @JvmStatic
 fun getPictureSize(file: File): Size {
     return if (file.exists()) {
         val opts = BitmapFactory.Options()
         // 不读取像素数组到内存中，仅读取图片的信息
         opts.inJustDecodeBounds = true
         BitmapFactory.decodeFile(file.path, opts)
         Size(opts.outWidth, opts.outHeight)
     } else {
         Size(0, 0)
     }
 }

动图播放 Movie

Movie 是Android本身提供的一个解析播放GIF类 与 Bitmap 处于同一个包下 android.graphics ，

简介

  1. 三个静态构造方法
    • public static Movie decodeStream(InputStream is)
    • public static native Movie decodeByteArray(byte[] data, int offset,int length)
    • public static Movie decodeFile(String pathName)
  2. 对外方法
    • public native int width()
    • public native int height()
    • public native boolean isOpaque();// 是否不透明
    • public native int duration();// Gif播放周期
    • public native boolean setTime(int relativeMilliseconds);// 设置时间戳 即可定位到指定的帧
    • public void draw(Canvas canvas, float x, float y, Paint paint);//在画布上画指定的帧
    • public void draw(Canvas canvas, float x, float y)

官方示例代码

//获取指定GIF的InputStream
is = context.getResources().openRawResource(R.drawable.animated_gif);
//构造
if (DECODE_STREAM) {
    mMovie = Movie.decodeStream(is);
} else {
    byte[] array = streamToBytes(is);
    mMovie = Movie.decodeByteArray(array, 0, array.length);
}

//绘制 onDraw
long now = android.os.SystemClock.uptimeMillis();
if (mMovieStart == 0) {   // first time
    mMovieStart = now;
}
if (mMovie != null) {
    int dur = mMovie.duration();
    //防止出现为0的情况
    if (dur == 0) {
        dur = 1000;
    }
    //在GIF周期内无限循环
    int relTime = (int)((now - mMovieStart) % dur);
    mMovie.setTime(relTime);
    //在canvas的右下角绘制
    mMovie.draw(canvas, getWidth() - mMovie.width(),
                getHeight() - mMovie.height());
    //触发onDraw 从而达到循环的效果            
    invalidate();
}
