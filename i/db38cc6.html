cache

我们用Git提交版本的时候，忽略文件有可能并不是不变的。但是对于后来加入到.gitignore中的文件，GIT默认还是不理睬它们的。因为GIT有一个缓存的机制。

$ git rm -r --cached .
$ git add .
$ git commit -m ".gitignore is now working"

revert

git revert是提交一个新的版本，将需要revert的版本的内容再反向修改回去，版本会递增，不影响之前提交的内容

$ git revert HEAD             撤销前一次 commit
$ git revert HEAD^            撤销前前一次 commit
$ git revert commit-id （比如：fa042ce57ebbe5bb9c8db709f719cec2c58ee7ff）撤销指定的版本，撤销也会作为一次提交进行保存。

rebase

git一般的流程是先拉取最新代码（git pull）,再向服务器提交自己的代码(git push),当然如果有冲突先解决冲突。但是执行git pull的时候会执行一个merge操作，这样从git log中看 git提交记录线会比较乱。这个时候rebase就派上用场了。git pull --rebase的作用是先把当前工作分支的代码同步成最新的 再merge我们修改的代码，如果有冲突需先修改冲突。修改完冲突 再 git rebase continue .然后再提交代码。

git pull --rebase
git rebase continue
git push

stash

git stash 应用的场景是，你的修改进行了一半，还不想提交记录。比如，你正在开发一个功能，还没开发结束，你还不能commit,这个时候经理告诉你，有个线上BUG需要先去处理。按照流程，你需要切换分支，但切换分支的前提是，你要commit 当前修改的内容。为了解决这个问题。git stash 派上用场了。git stash 会暂存你的修改记录，存进一个stash目录里面。当你处理完别的任务，回来的时候执行下 git stash apply 就可以继续开发工作了。

git stash
git stash apply

git remote

作用，设置多个代码源；在不同代码源更新或推送代码。常见场景为 github fork 代码后 及时更新源项目代码

# 查看代码源
$ git remote -v
# 查看 remote 相关帮助
$ git remote --help
# 添加新源
$ git remote add  dfl git@github.com:iperov/DeepFaceLab.git
# 移除源
$ git remote remove dfl
# 从 dfl master 分支更新并合并代码
$ git pull dfl master --rebase

更新本地branch

$ git remote update origin --prune
$ git remote show origin
$ git remote prune origin

使用系统别名

比如想实现 Android Studio 中拉取代码的功能,可以

$ vim ~/.bash_profile
# 在 .bash_profile 中最后一行添加 alias gps="git stash save xx && git pull --rebase && git stash pop"
$ source ~/.bash_profile
# 这样只需要在终端输入  gps 即可拉取代码了
